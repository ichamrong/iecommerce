# Module Specification: Subscription (SaaS Billing & Plans)

## 1. Purpose
The Subscription module is the core of the SaaS business model. It manages the billing relationship between the **SaaS Platform Owner** and the **Merchants (Shop Owners, Hoteliers, Service Providers)**. It enforces plan limits, free trials, and feature access.

## 2. Core Domain Models
- **SubscriptionPlan**: Defines offering tiers (e.g., `ECOMMERCE_BASIC`, `ECOMMERCE_PRO`, `BOOKING_STANDARD`, `ACCOMMODATION_PREMIUM`).
  - **Pricing**: Monthly/Yearly cost.
  - **Trial Period**: Number of days for a free trial.
  - **Features**: A JSON map of enabled features (e.g., `{"canCreatePhysicalProducts": true, "canCreateBookings": false}`).
- **TenantSubscription**: The active plan assigned to a specific `tenant_id`. 
  - **Status**: `TRIALING`, `ACTIVE`, `PAST_DUE`, `CANCELED`.
  - **BillingCycle**: Next invoice date.
- **SaaSInvoice**: Bills generated by the SaaS platform and sent to the Merchant (Shop Owner) for their subscription.

## 3. The B2B2C Ecosystem (Relationship Model)
This platform operates on a **B2B2C (Business-to-Business-to-Consumer)** model. It's critical to distinguish between the two distinct "billing" and "service" relationships that exist on the platform:

### Tier 1: SaaS Admin â†” Shop Admin (B2B)
- **SaaS Admin (You):** The platform owner. You build, host, and maintain the API and underlying infrastructure.
- **Shop Admin (Merchant):** The shop owner, hotelier, or service provider. They are **your customer**.
- **The Transaction:** The Shop Admin pays the SaaS Admin a **subscription fee** (monthly/yearly) to use the platform. They might buy the `ECOMMERCE_PRO` plan or the `BOOKING_STANDARD` plan.
- **The Deliverable:** You provide them with login access to a "Super Admin" dashboard where they can manage their specific `tenant_id` sandbox (add products, view reports). You also provide them access to the Headless API so they can power their storefronts.

### Tier 2: Shop Admin â†” End Customer (B2C)
- **End Customer:** The person actually buying a t-shirt, booking a massage, or reserving a hotel room.
- **The Transaction:** The End Customer pays the **Shop Admin** for their goods or services.
- **The Deliverable:** The End Customer interacts with the storefront (which might be the default storefront you provide the Shop Admin, or a custom one built by the Shop Admin's developers). They don't know the SaaS Admin exists. They only see the Shop Admin's brand.
- **Multi-Tenancy:** End Customers belong to the Shop Admin. A customer registered on "Shop A" has no access to "Shop B" unless they register there as well.

## 4. Key Business Logic
- **Feature Gating**: Before a merchant can use specific endpoints (like creating a Room Type or a Physical Product), the API Gateway or Interceptor checks the `TenantSubscription` to verify they have paid for that service paradigm.
- **Trial Expiration**: A daily cron job checks for expired trials. If a trial expires without a payment method on file, the tenant's `Status` shifts to `PAST_DUE` and their public storefront APIs are temporarily suspended.
- **Plan Upgrades/Downgrades**: Merchants can switch from an E-commerce plan to a dual E-commerce + Booking plan. Changes are prorated.

## 5. Multi-Tenancy Strategy (SaaS)
- This module operates primarily at the "Super Admin" level. 
- Individual tenants (merchants) can read their own `TenantSubscription` to see their limits, but cannot modify the overarching `SubscriptionPlan` definitions.

## 6. Public APIs (Internal Modulith)
- `SubscriptionService.getActivePlan(tenantId)`: Evaluates what features a shop owner is currently allowed to access.
- `SubscriptionService.checkQuota(tenantId, featureKey)`: Enforces soft/hard limits (e.g., max 100 products on basic plan).
- `SubscriptionService.subscribe(tenantId, planId)`: Initiates the checkout process for a merchant starting a SaaS plan.
